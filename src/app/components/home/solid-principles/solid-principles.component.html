SOLID Principles

S

Example: A chef should only cook, not also be the waiter, cashier, and dishwasher.

// BAD - Does everything
class Employee {{ '{' }}
    void cook() {{ '{' }} /* ... */ {{ '}' }}
    void serve() {{ '{' }} /* ... */ {{ '}' }}
    void cashier() {{ '{' }} /* ... */ {{ '}' }}
{{ '}' }}

// GOOD - Single responsibility
class Chef {{ '{' }}
    void cook() {{ '{' }} /* only cooking */ {{ '}' }}
{{ '}' }}
Single Responsibility

A class should have only one reason to change

O

Example: A smartphone case protects your phone (closed) but you can add accessories like rings or stands (open for extension).

// BAD - Modifying existing class
class PhoneCase {{ '{' }}
    void protect() {{ '{' }} /* ... */ {{ '}' }}
    void addRing() {{ '{' }} /* added later! */ {{ '}' }}
{{ '}' }}

// GOOD - Extend without modifying
interface Accessory {{ '{' }}
    void attach();
{{ '}' }}

class RingStand implements Accessory {{ '{' }}
    void attach() {{ '{' }} /* new functionality */ {{ '}' }}
{{ '}' }}
Open/Closed

Open for extension, closed for modification

L

Example: If you ask for a "bird" to fly, don't give me a penguin! Any bird substitute should be able to do what a bird does.

// BAD - Penguin can't fly!
class Bird {{ '{' }}
    void fly() {{ '{' }} /* ... */ {{ '}' }}
{{ '}' }}

class Penguin extends Bird {{ '{' }}
    void fly() {{ '{' }} throw new Error("Can't fly!"); {{ '}' }}
{{ '}' }}

// GOOD - Consistent behavior
abstract class Bird {{ '{' }}
    abstract void move();
{{ '}' }}

class Penguin extends Bird {{ '{' }}
    void move() {{ '{' }} /* swim */ {{ '}' }}
{{ '}' }}
Liskov Substitution

Child classes must be replaceable with their parent without breaking code

I

Example: A TV remote shouldn't force you to have airplane controls. Give specific tools for specific jobs.

// BAD - TV remote with airplane controls
interface Remote {{ '{' }}
    void changeChannel();
    void adjustAltitude(); // Not needed for TV!
{{ '}' }}

// GOOD - Specific interfaces
interface TVRemote {{ '{' }}
    void changeChannel();
{{ '}' }}

interface AirplaneControls {{ '{' }}
    void adjustAltitude();
{{ '}' }}
Interface Segregation

Many specific interfaces are better than one general

D

Example: A light switch works with any bulb (LED, incandescent, fluorescent) because it depends on the concept of "lightable" not a specific bulb type.

// BAD - Light switch directly controls specific bulb
class LightSwitch {{ '{' }}
    LEDBulb bulb = new LEDBulb();
    void toggle() {{ '{' }} bulb.turnOn(); {{ '}' }}
{{ '}' }}

// GOOD - Light switch works with any bulb
class LightSwitch {{ '{' }}
    Bulb bulb; // Depends on abstraction
    void toggle() {{ '{' }} bulb.illuminate(); {{ '}' }}
{{ '}' }}

interface Bulb {{ '{' }} void illuminate(); {{ '}' }}
class LEDBulb implements Bulb {{ '{' }} /* ... */ {{ '}' }}
class CFLBulb implements Bulb {{ '{' }} /* ... */ {{ '}' }}
Dependency Inversion

Depend on abstractions, not concretions
